# プロジェクトの目的・完成形

> `lora-workbench`。Illustrious (SDXL) 向けのキャラ／画風 LoRA を、
> **データセット準備から学習・即時テスト・比較評価まで一気通貫**で回せる
> ローカルデスクトップアプリ。詳細仕様は [`docs/SPEC.md`](../SPEC.md)。

## 1. 目的

参照画像をドロップするだけで、次の一連の流れを **一つのアプリ内** で回せること。

```
Import → 自動タグ付け → タグ剪定 → データセット構築 → 学習 → 生成テスト → 比較評価
```

「学習して終わり」ではなく、**学習直後に実運用モデルで生成して、採用する
epoch / weight をその場で決める**ところまでを閉じたループにするのが主眼。

## 2. 完成形（何ができれば完成か）

- 参照画像を raw にドロップ → 重複除去・前提チェックが自動で走る
- WD14 で客観タグ、（任意で）VLM で自然文キャプションが付く
- **タグ頻度分析＋テキスト LLM による剪定**で、キャラ固有タグをトリガーワードへ
  焼き込み、状況依存タグは残す。人がレビューできる（human-in-the-loop）
- repeats 設定から kohya 入力形式（`train/{repeats}_{concept}/` ＋ `.txt`）を構築
- Illustrious 向けデフォルトの学習 TOML を生成
- kohya_ss を subprocess 起動し、ログを配信・epoch ごとに LoRA とサンプルを取得
- WebUI Forge で「固定プロンプト × epoch × weight × 固定 seed」のグリッド生成
- 比較ギャラリーで採用する epoch / weight を決定できる

## 3. 重視する価値

- **車輪の再発明をしない**（設計の芯）。既存の実績あるツールを疎結合でラップする。
  - 学習エンジンは自作しない → **kohya_ss (sd-scripts)** を subprocess でラップ
  - 推論エンジンは自作しない → **WebUI Forge** の `/sdapi/v1` をラップ
  - タガー／VLM は自作しない → **WD14 (onnxruntime)** と **llama.cpp** をラップ
- **backend はオーケストレーションに徹する**。重い処理はすべて独立プロセス
  （HTTP / subprocess）として疎結合にし、backend は起動・監視・呼び出しのみ。
- **品質の 8 割はデータセットで決まる**。タグ付け・剪定パイプラインを最優先で固める。
- **ローカル完結**。クラウド学習、課金、マルチユーザ、リモート配信は対象外。

## 4. 非ゴール

- 学習／推論エンジンの内製
- クラウド学習・課金・マルチユーザ・リモート配信
- diffusers を backend に抱えること（推論は Forge に委譲）

## 5. ローカル資産の配置方針

- **ローカル LLM モデル（VLM / テキスト LLM）** は `models/` フォルダへダウンロードして配置。
- **外部エンジン**は `runtime/` フォルダ配下へダウンロード・インストール（engine ごとにサブフォルダ）。
  - `runtime/llama.cpp/` — 推論(LLM)サーバー
  - `runtime/kohya_ss/` — 学習エンジン（専用 venv は `runtime/kohya_ss/.venv`）
  - `runtime/forge/` — WebUI Forge（生成テスト）
- どちらもマシン依存・大容量のため **git 管理外**（`.gitignore`）とする。
